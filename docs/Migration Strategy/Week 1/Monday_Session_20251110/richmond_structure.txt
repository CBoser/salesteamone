EXECUTIVE SUMMARY
================================================================================

Key Findings:
1. **Plan Sheets**: Each plan has its own worksheet with materials organized
   by pack and elevation
   
2. **Elevation Encoding**: TRIPLE-ENCODED in location string
   - Example: "|10 FOUNDATION - ELVA - ELVB - ELVC" encodes:
     * Phase: 10 FOUNDATION
     * Elevations: A, B, C (as ELVA, ELVB, ELVC)
   
3. **Pack System**: Hierarchical pack IDs with decimal structure
   - Base: |10 FOUNDATION
   - Options: |10.01, |10.60x, |10.82, |10.82BCD
   - Suffixes indicate variants (x=extension, tc=tall crawl, etc.)

4. **Option Codes**: Richmond option codes separate from pack IDs
   - Pack: |10.01 OPT FOUNDATION
   - Option code: FPSING01 (fireplace single)
   
5. **Critical Issue**: Pack 10.82 vs 10.82BCD demonstrates elevation encoding
   problem that needs to be solved in database design

PLAN NAMING PATTERNS
================================================================================

Three distinct naming patterns observed:

PATTERN 1: Plan Number Only (Base)
  Examples: G603, G914
  Format: [Plan Number]
  Usage: Multi-elevation plans with all variants included
  Elevation Handling: All elevations (A, B, C) in one sheet

PATTERN 2: Community Code + Plan + Elevation
  Examples: LE93 G603B, LE94 G603A
  Format: [Community Code] [Plan Number][Elevation Letter]
  Where: LE = Luden Estates, number = lot/variant
  Usage: Specific lot or variant in a community
  Elevation Handling: Single elevation per sheet

PATTERN 3: Plan + Elevation + Community Abbreviation
  Examples: 1649A PTO GG, 1890ABD CR, 2299A PTO GG
  Format: [Plan Number][Elevations] [Descriptor] [Community]
  Where: 
    - PTO = "Parr Takeoff"
    - CR = Creek Ridge
    - GG = Golden Grove
  Usage: Community-specific plans with descriptors
  Elevation Handling: Can be single (A) or multiple (ABD)

FINDING: No consistent plan naming standard
  - Some sheets = one elevation
  - Some sheets = all elevations
  - Community codes optional (GG, CR vs. base plans)
  - Lot numbers sometimes included (LE93, LE94)

ELEVATION ENCODING SYSTEM
================================================================================

Richmond uses a PIPE + PHASE + ELEVATION pattern:

Format: |[PhaseNumber] [PHASE NAME] - [ELV Codes]

Examples:
  |10 FOUNDATION - ELVA - ELVB - ELVC
  |11 MAIN JOIST SYSTEM - ELVA - ELVB - ELVC
  |20 MAIN WALLS READYFRAME - ELVA - ELVB - ELVC
  |18 MAIN FLOOR SUBFLOOR - ELVA - ELVB - ELVC

Elevation Code Structure:
  ELV + [Letter] = Elevation [Letter]
  
  ELVA = Elevation A
  ELVB = Elevation B
  ELVC = Elevation C
  ELVD = Elevation D

Multiple Elevations:
  When a material applies to multiple elevations, ALL elevation codes appear:
  "- ELVA - ELVB - ELVC" = applies to elevations A, B, and C

Single Elevation:
  When specific to one elevation, only that code appears:
  "- ELVB" = applies only to elevation B

CRITICAL FINDING: TRIPLE ENCODING PROBLEM

Location: |10.82BCD OPT DEN FOUNDATION - ELVB - ELVC - ELVD

This encodes elevation THREE times:
  1. In pack ID: "10.82BCD" (BCD = elevations B, C, D)
  2. In location string: "- ELVB - ELVC - ELVD"
  3. In option codes: ELVB, ELVC, ELVD (from reference sheet)

This creates maintenance problems:
  - Must update in 3 places if elevation changes
  - Risk of inconsistency
  - Violates DRY (Don't Repeat Yourself) principle

RECOMMENDATION: Database should store elevation ONCE
  - Pack definition: "10.82 OPT DEN FOUNDATION"
  - Elevation availability: B, C, D (in separate field)
  - Job selection: Which elevation customer chose

PACK ID SYSTEM
================================================================================

Richmond uses hierarchical pack IDs with this structure:

Format: |[Major].[Minor][Suffix] [PACK NAME] - [Option Code]

Components:

1. PIPE (|): Separates pack sections, indicates hierarchy level
   
2. MAJOR NUMBER (10, 11, 20, etc.): Construction phase
   10 = Foundation
   11 = Floor system / Joist
   12 = (likely garage or specific area)
   18 = Subfloor
   20 = Walls
   [Full phase list not yet documented]

3. MINOR NUMBER (.01, .60, .82, etc.): Variant within phase
   - Sequential numbering for options
   - No clear pattern (01, 60, 61, 62, 82, 83, etc.)

4. SUFFIX (optional):
   x  = extension (60x = extended great room)
   tc = tall crawl (10.tc = tall crawl framing)
   BCD = elevation restriction (built into pack ID - problematic!)
   b = bath variant (83b vs 82 = den with bath vs den alone)

5. PACK NAME: Descriptive text
   Examples:
   - "FOUNDATION"
   - "OPT DEN FOUNDATION"
   - "EXTENDED GREAT ROOM FOUNDATION"
   - "MAIN JOIST SYSTEM"

6. OPTION CODE (after dash): Richmond's external option code
   Examples:
   - FPSING01 (fireplace single)
   - XGREAT (extended great room)
   - SUN (sunroom)
   - TALLCRWL (tall crawl)

CRITICAL ARCHITECTURAL DECISIONS NEEDED
================================================================================

Based on Richmond structure analysis, these decisions must be made:

DECISION 1: Plan-Elevation Model
  
  Question: Is "G603B" one plan or is it Plan G603 + Elevation B?
  
  Evidence:
    - G603 sheet contains elevations A, B, C (all in one)
    - LE93 G603B sheet is elevation B only
    - Some packs show "|10BCD" vs "|10" (elevation in pack ID)
  
  Options:
    A) Elevation as Plan Variant
       - G603, G603B, G603C are separate plans
       - plan_id = "G603B" (elevation embedded)
    
    B) Elevation as Dimension
       - G603 is the plan
       - B is an elevation of G603
       - plan_id = "G603", elevation = "B"
  
  Implication: Affects database schema design

DECISION 2: Pack-Elevation Relationship
  
  Question: How to handle packs like "10.82" vs "10.82BCD"?
  
  Current approach: Elevation suffix in pack ID
    - |10.82 OPT DEN FOUNDATION (no elevation restriction?)
    - |10.82BCD OPT DEN FOUNDATION (only B, C, D)
  
  Problems:
    - Creates duplicate pack definitions
    - Triple-encodes elevation data
    - Hard to query "show me all den options"
  
  Options:
    A) Keep elevation in pack ID
       - pack_id = "10.82BCD"
       - Matches current Richmond approach
    
    B) Separate pack from elevation restriction
       - pack_id = "10.82"
       - elevation_availability = "B, C, D"
       - Cleaner, avoids duplication
  
  Implication: Affects how materials are queried and filtered

DECISION 3: Option Code Philosophy
  
  Question: What's the relationship between pack ID and option code?
  
  Current approach:
    - Pack: |10.60x EXTENDED GREAT ROOM FOUNDATION
    - Option: XGREAT
    - Two separate identifiers for same thing
  
  Options:
    A) Keep both (translation layer)
       - Pack ID = internal organization (10.60x)
       - Option code = customer-facing (XGREAT)
       - Map between them in database
    
    B) Unify to one code
       - Use option code as primary (XGREAT)
       - OR use pack ID as primary (10.60x)
       - Eliminate redundancy
  
  Implication: Affects how options are quoted and tracked
