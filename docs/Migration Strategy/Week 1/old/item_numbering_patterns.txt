ITEM NUMBERING PATTERNS ANALYSIS
================================================================================
Date: November 10, 2025
Analyzed: Richmond 3BAT and Holt BAT systems
Purpose: Establish baseline understanding of item coding systems before
         designing unified coding standard

================================================================================
EXECUTIVE SUMMARY
================================================================================

Total Items Analyzed:
  - Richmond:  633 unique SKUs
  - Holt:      113 unique SKUs (from IWP/RL pricing sheets)

Key Findings:
  1. DIFFERENT SYSTEMS: Richmond and Holt use completely different item 
     numbering approaches with minimal overlap
  
  2. Richmond is SKU-based: Alphanumeric codes (e.g., 2616HF3TICAG, 12GL312T)
     Holt uses descriptions: Plain text (e.g., "2X4 STD&BTR HEM/FIR KD")
  
  3. No consistent prefix/suffix system across builders
  
  4. High variability in Richmond SKUs suggests multiple vendor sources or
     historical code systems that have been merged

COMPARISON: RICHMOND vs HOLT
================================================================================

Total Items:
  Richmond:    633 unique SKUs
  Holt:        113 unique items (60 coded SKUs + 43 descriptions + 10 notes)

-------------------------------------------
FUNDAMENTAL DIFFERENCES
-------------------------------------------

1. CODING PHILOSOPHY:
   Richmond: Vendor SKU passthrough
             - Uses actual vendor/manufacturer part numbers
             - High variability, multiple encoding systems
             - SKUs from Simpson, lumber mills, distributors
   
   Holt:     Descriptive with some internal codes
             - Mix of internal codes (DFKDR series) and plain descriptions
             - Plain English for commodity lumber ("2X4 STD&BTR HEM/FIR KD")
             - More human-readable

2. CONSISTENCY:
   Richmond: Low consistency
             - 288 different prefix patterns across 633 items
             - No dominant prefix (largest is 2.7%)
             - Reflects multiple vendors/sources
   
   Holt:     High consistency in timber codes
             - DFKDR/DFKDS = 53% of all items
             - Systematic dimensional encoding
             - Standardized descriptive format for commodity items

3. STRUCTURE:
   Richmond: NUMBER-LETTER dominant (38.6%)
             - Example: 2616HF3TICAG
             - Encodes: dimension, species, grade, treatment
   
   Holt:     LETTER-NUMBER-LETTER dominant (53.1%)
             - Example: DFKDR26XX
             - Encodes: species, treatment, dimension, length placeholder

4. SPECIAL CHARACTERS:
   Richmond: Minimal (mostly hyphens)
             - Clean alphanumeric codes
             - Special chars in only 9% of items
   
   Holt:     Moderate (fractions and measurements)
             - Inches, fractions in descriptions
             - Special chars in 30% of items

5. LENGTH ENCODING:
   Richmond: Embedded in SKU
             - Not consistently encoded
             - Mix of specific lengths and "any length"
   
   Holt:     XX placeholder system
             - DFKDR24XX means "fill in length"
             - Or specific in descriptions: "92-5/8\" STUD"

================================================================================
RECOMMENDATIONS FOR UNIFIED CODING SYSTEM
================================================================================

Based on analysis of 746 unique items across both systems:

-------------------------------------------
1. PRESERVE EXISTING SKUs
-------------------------------------------

RECOMMENDATION: Use vendor SKUs as primary identifier, add internal mapping layer

Rationale:
  - Richmond already uses vendor SKUs (633 items)
  - Changing SKUs breaks vendor integrations and purchase orders
  - Historical data and pricing tied to existing SKUs
  - Team familiarity with current codes

Approach:
  - SKU field = exact vendor part number (preserve Richmond pattern)
  - For Holt descriptive items: create internal SKU or use description as key
  - Add separate internal_code field for cross-referencing

-------------------------------------------
2. STANDARDIZE DESCRIPTIONS
-------------------------------------------

RECOMMENDATION: Separate SKU from description, make descriptions consistent

Richmond Issue: Some SKUs are full descriptions ("16x60 vinyl shutter louvered")
Holt Pattern: Clean separation between code and description

Standard Format:
  SKU:          2616HF3TICAG
  Description:  2x6x16 Hem Fir #3 Treated Incised Ground Contact
  
  SKU:          DFKDR26XX
  Description:  2x6 Doug Fir Kiln Dried Resawn, Length Variable

-------------------------------------------
3. CATEGORY CLASSIFICATION
-------------------------------------------

RECOMMENDATION: Add category field with consistent hierarchy

Proposed Categories:
  10 - DIMENSIONAL LUMBER
       10.1 - Doug Fir
       10.2 - Hem Fir
       10.3 - Other Species
  
  20 - ENGINEERED LUMBER
       20.1 - I-Joists (TJI)
       20.2 - LVL/LSL
       20.3 - Glulam
  
  30 - SHEATHING
       30.1 - OSB
       30.2 - Plywood
  
  40 - STRUCTURAL HARDWARE
       40.1 - Hangers
       40.2 - Connectors
       40.3 - Fasteners
  
  50 - SPECIALTY
       50.1 - Pressure Treated
       50.2 - Architectural Elements
       50.3 - Custom Items

Rationale: Enables filtering, reporting, price schedule organization

-------------------------------------------
4. ATTRIBUTE TAGGING
-------------------------------------------

RECOMMENDATION: Extract attributes into separate searchable fields

Standard Attributes:
  - species: DF, HF, SPF, WRC, etc.
  - treatment: KD, PT, None
  - grade: #1, #2, STD, SELECT, etc.
  - nominal_size: 2x4, 2x6, 4x4, etc.
  - actual_size: 1.5x3.5, 1.5x5.5, etc.
  - length: specific or "random" or "variable"

Benefits:
  - Searchable by attribute
  - Enables quantity takeoffs by dimension
  - Facilitates price comparisons
  - Supports future web interface

-------------------------------------------
5. PREFIX STANDARDIZATION
-------------------------------------------

RECOMMENDATION: Do NOT try to unify prefixes - keep vendor SKUs intact

Why:
  - 288 different prefix patterns in Richmond alone
  - Prefixes tied to vendor/manufacturer systems
  - Attempting to unify would create massive data migration
  - Purchase orders and vendor systems expect specific SKUs

Instead:
  - Use category codes (10, 20, 30, etc.) for internal organization
  - Map vendor SKUs to internal categories
  - Enable searching by category OR SKU

-------------------------------------------
6. LENGTH HANDLING
-------------------------------------------

RECOMMENDATION: Adopt Holt's XX placeholder system for variable-length items

Two item types:
  1. Specific length: 2616HF3TICAG (2x6x16 HF treated)
  2. Variable length: 26HF3TICAG-XX or 26HF3TICAG-VAR

Benefits:
  - One SKU per product specification
  - Length specified at order time, not at item definition time
  - Reduces SKU proliferation
  - Matches Holt pattern (DFKDR26XX)

-------------------------------------------
7. COMMUNITY/ELEVATION HANDLING
-------------------------------------------

RECOMMENDATION: Keep community and elevation OUT of item SKUs

Confirmed: Neither Richmond nor Holt currently encode these in SKUs
Continue this pattern:
  - Item SKU = product identifier (what it is)
  - Community = job attribute (where it's used)
  - Elevation = plan attribute (which variant)

Relationship:
  Job → Plan → Elevation → Material List → Items (by SKU)
  
Community and elevation are contexts for SKU usage, not part of SKU identity.
